# Участие в разработке — Contractor Frontend

## Модель ветвления Git

- **Ветка по умолчанию:** `main` (или `master`). Сюда попадает код, готовый к продакшену.
- **Фичи:** создавать ветку от `main` (напр. `feature/dashboard-filters`, `fix/echo-reconnect`). Долгоживущей ветки разработки нет, если проект явно её не вводит.
- **Staging (опционально):** при использовании ветка `staging` обновляется из `main` или из фича-веток для предпродакшен-тестов; описать в DEPLOYMENT.md.

## Соглашение по именованию веток

| Префикс | Назначение | Пример |
|---------|------------|--------|
| `feature/` | Новая фича или доработка | `feature/contract-export` |
| `fix/` | Исправление бага | `fix/auth-redirect-loop` |
| `docs/` | Только документация | `docs/setup-env-table` |
| `chore/` | Инструменты, зависимости, конфиг | `chore/upgrade-next` |

После префикса — kebab-case. Имена короткие и понятные.

## Требования к Pull Request

1. **Цель:** открывать PR в `main` (или в ветку по правилам проекта).
2. **Объём:** одно логическое изменение на PR; по возможности дробить крупные задачи.
3. **Описание:** что изменено и зачем; при наличии — ссылка на issue или ADR.
4. **Проверки:** CI (lint, build) должен проходить; замечания ревью закрыть до мержа.
5. **Без force-push** в целевую ветку после начала ревью, если команда не договорилась об обратном.

## Чек-лист ревью (для ревьюеров)

| Пункт | Проверить |
|-------|-----------|
| **Архитектура** | Соответствие маршрутам, компонентам и использованию lib; без новых ad-hoc паттернов без ADR |
| **Использование API** | Используется `lib/api.ts`; нет сырого fetch к API |
| **Типы** | Типы TypeScript для нового API или пропсов; без неоправданного `any` |
| **Авторизация** | Защищённые маршруты по-прежнему под охраной; токен не в URL и не в логах |
| **Ошибки** | Состояния загрузки и ошибки для асинхронных потоков |
| **Тесты** | Новая логика покрыта тестами по TESTING.md |
| **Документация** | README или setup/docs обновлены при изменении поведения или env |
| **Линт** | ESLint и TypeScript проходят |

## Стандарт сообщений коммитов

- **Формат:** короткая строка (≤72 символа); при необходимости тело и футер.
- **Время:** повелительное наклонение («Добавить фильтр», «Исправить редирект»), не прошедшее («Добавлено», «Исправлено»).
- **Scope (опционально):** префикс области, напр. `dashboard: add empty state`, `api: fix token attach`.

Примеры:

```
feature: add contract list filter by status
fix: prevent double submit on analyse form
docs: document NEXT_PUBLIC_REVERB_* in ENVIRONMENT.md
```

## Политика слияния

- **Кто мержит:** после минимум одного апрува (или по правилам проекта); мержить может автор или ревьюер.
- **Squash vs merge commit:** по предпочтениям проекта; история должна оставаться понятной.
- **Удаление ветки:** удалять фича-ветку после мержа.

## Обязательные проверки

Перед мержем должны проходить (как правило в CI):

| Проверка | Команда | Назначение |
|----------|---------|------------|
| Lint | `npm run lint` | ESLint |
| Build | `npm run build` | TypeScript и сборка Next.js |

Дополнительные проверки (unit, E2E) описаны в [TESTING.md](TESTING.md) и включаются в CI по мере появления.

| ☐ | Документация обновлена при изменении setup или env |
