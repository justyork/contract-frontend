# Тестирование — Contractor Frontend

## Уровни тестирования

| Уровень | Область | Инструменты / подход | Когда использовать |
|---------|---------|----------------------|--------------------|
| **Unit** | Функции, хуки, чистая логика | Jest (или Vitest), React Testing Library | Утилиты, форматтеры, хуки без тяжёлого DOM |
| **Компонент** | React-компоненты изолированно | React Testing Library, Jest | UI-компоненты с пропсами и действиями пользователя |
| **Интеграция** | Несколько модулей (API-клиент + хук + компонент) | Jest, MSW (Mock Service Worker) | Потоки, пересекающие lib и UI |
| **E2E** | Полное приложение в браузере | Playwright или Cypress | Критические пути: логин, анализ, просмотр результата |

Не все уровни могут быть реализованы; добавлять тесты по мере внедрения стека. Документ задаёт целевую стратегию и правила.

## Правила unit-тестов

- **Чистые функции:** тестировать прямым вызовом; без DOM. Покрывать граничные случаи (пусто, null, невалидный ввод).
- **Хуки:** использовать `@testing-library/react-hooks` или рендер маленького обёрточного компонента и проверять состояние/эффекты.
- **Без деталей реализации:** тестировать поведение (результат, побочные эффекты), а не внутреннее состояние или приватные функции.
- **Расположение:** рядом с модулем в `*.test.ts` / `*.test.tsx` или в папке `__tests__` рядом с фичей.

## Правила интеграционных тестов

- **Поверхность API:** мокать на уровне сети (MSW) или мокать `lib/api.ts`; в CI не вызывать реальный API.
- **Echo:** мокать клиент Echo или эмиссию событий; проверять, что UI обновляется при «получении» событий.
- **Авторизация:** мокать контекст авторизации или токен в localStorage для защищённых потоков.
- **Один поток на файл (или чёткие describe):** напр. «поток анализа: отправка → прогресс → результат».

## Стратегия E2E

- **Критические пути:** логин (или auth callback), открытие дашборда, запуск анализа, просмотр результата (или прогресса). Добавить платёжный поток по мере стабилизации.
- **Окружение:** отдельное тестовое окружение или заглушка API; не гонять против продакшена.
- **Стабильность:** предпочитать data-testid или селекторы по ролям; избегать хрупких CSS-селекторов. Ждать по сети или видимому состоянию, а не фиксированным таймаутам.
- **CI:** запускать E2E в CI по расписанию или при мерже в main; описать в CONTRIBUTING и DEPLOYMENT.

## Требования к покрытию

| Область | Цель | Примечания |
|---------|------|------------|
| `lib/` (api, echo, utils) | Высокое | Ядро логики; стремиться к >80% где возможно |
| Хуки | Высокое | Аналогично lib |
| Компоненты | Среднее | Минимум критические и общие компоненты |
| Страницы | Ниже | Покрываются косвенно E2E или интеграцией |
| Типы | N/A | Покрытия в рантайме нет |

Покрытие включать в CI, когда раннер поддерживает (напр. Jest `--coverage`). Не писать тесты ради цифры; приоритет — осмысленные сценарии.

## Правила моков

- **API:** мокать `fetch` или использовать MSW handlers; формы ответов согласовать с `types/api.ts`.
- **Echo:** мокать клиент Echo или модуль подписки; эмитировать события и проверять UI или состояние.
- **Роутер:** использовать утилиты Next.js для тестов или мокать `useRouter` / `usePathname` при проверке навигации.
- **Окружение:** при необходимости мокать `process.env.NEXT_PUBLIC_*` в тестах; в CI не полагаться на реальное окружение.
- **Не мокать детали реализации:** мокать на границах (сеть, роутер, контекст).

## Соглашение по именованию тестов

- **Файл:** `*.test.ts`, `*.test.tsx` или `*.spec.ts` (выбрать один вариант и придерживаться).
- **Describe:** имя компонента или модуля, либо имя потока. Пример: `describe("ContractSummaryCard", () => { ... })`.
- **It / test:** поведение в настоящем времени. Пример: `it("показывает название контракта и кнопку просмотра", () => { ... })`.
- **Избегать:** размытых формулировок вроде «работает» или «test 1».

## Требования в CI

- **Линт и сборка:** должны проходить (см. CONTRIBUTING).
- **Unit и интеграция:** запускать на каждый PR; при падении — падать пайплайн.
- **E2E:** запускать при мерже в main или по расписанию; опционально на каждый PR при медленности или нестабильности.
- **Покрытие:** падать или предупреждать при пороге ниже заданного (напр. 70% для `lib/`); настраивать в Jest (или Vitest) и CI.

## Пример: unit-тест (утилита)

```ts
// lib/markdown.test.ts
import { renderMarkdown } from "./markdown";

describe("renderMarkdown", () => {
  it("returns empty string for empty input", () => {
    expect(renderMarkdown("")).toBe("");
  });
  it("renders bold and links", () => {
    expect(renderMarkdown("**a** [b](c)")).toContain("<strong>");
  });
});
```

## Пример: тест компонента

```tsx
// components/ui/Button.test.tsx
import { render, screen } from "@testing-library/react";
import userEvent from "@testing-library/user-event";
import { Button } from "./Button";

describe("Button", () => {
  it("renders label and calls onClick when clicked", async () => {
    const onPress = jest.fn();
    render(<Button onClick={onPress}>Submit</Button>);
    await userEvent.click(screen.getByRole("button", { name: /submit/i }));
    expect(onPress).toHaveBeenCalledTimes(1);
  });
});
```
