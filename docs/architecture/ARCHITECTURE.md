# Архитектура — Contractor Frontend

## Стиль архитектуры

Contractor Frontend — **SPA-приложение на Next.js App Router** с чётким разделением:

1. **Маршруты и лейауты** — `app/` задаёт структуру URL и общие лейауты.
2. **UI-компоненты** — `components/` для переиспользуемых презентационных и фичевых компонентов.
3. **Логика приложения** — хуки, `lib/` (API, Echo, утилиты) и `contexts/` для общего состояния.
4. **Типы** — общие типы TypeScript для контрактов API и компонентов.

Стиль: **компонентный + тонкий слой данных**: страницы собирают компоненты и вызывают единый API-клиент и Echo; по умолчанию без Redux и глобального стора.

## Определение слоёв

### Слой маршрутов / лейаутов

**Расположение:** `src/app/`

**Ответственность:**
- Определение публичных и защищённых маршрутов.
- Корневой и вложенные лейауты (напр. `app/layout.tsx`, `app/ai/layout.tsx`).
- Монтирование страниц и передача состояния уровня layout (напр. авторизация) через контекст или пропсы.
- Проверка авторизации для `/ai/*` (через layout или `ProtectedRoute`).

**Правила:**
- Страницы не содержат бизнес-логику; они собирают компоненты и вызывают хуки.
- Сегменты маршрутов соответствуют путям URL; без скрытой логики маршрутизации.
- Лейауты владеют общей оболочкой (shell, навигация); страницы — контентом страницы.

**Пример структуры:**
```
app/
├── layout.tsx           # Корневой layout, шрифты, провайдеры
├── page.tsx             # Главная (публичная)
├── ai/
│   ├── layout.tsx       # Оболочка для авторизованных (AiShell, сайдбар, топбар)
│   ├── dashboard/
│   ├── analyse/
│   ├── contract/[id]/
│   ├── tokens/
│   └── profile/
├── auth/callback/
├── login/, register/
├── payment/success/, payment/cancel/
└── privacy/, terms/, cookie-policy/
```

### Слой компонентов

**Расположение:** `src/components/`

**Ответственность:**
- Отображение UI и обработка действий пользователя.
- Получение данных через пропсы или хуки; по возможности без прямых вызовов API в презентационных компонентах.
- Вызов колбэков для родителя или контейнера.
- Переиспользование на разных страницах.

**Подразделы:**
- `ui/` — примитивы (Button, Card, Container, Spinner, Badge и т.д.).
- `ai/` — оболочка приложения и дашборд (AiShell, AiSidebarNav, AiTopbar, ContractSummaryCard, EmptyState, PageHeader).
- `contract-analysis/` — отображение анализа (секции, оценки, навигация, markdown).
- `public/` — оболочка и футер публичных страниц.

**Правила:**
- Компоненты не импортируют из `app/`; их использует app.
- Предпочтительна композиция, а не прокидывание пропсов; контекст только для действительно общеприложенного состояния (напр. авторизация).
- Компоненты тестируемы за счёт явных пропсов и минимума побочных эффектов.

### Слой приложения / данных

**Расположение:** `src/lib/`, `src/contexts/` и хуки (в компонентах или в будущем `src/hooks/`)

**Ответственность:**
- **API:** единая точка входа `lib/api.ts` для всех REST-вызовов; использует `NEXT_PUBLIC_API_URL` и токен авторизации.
- **Echo:** `lib/echo.ts` для WebSocket и подписки на каналы (напр. прогресс анализа контракта).
- **Авторизация:** `contexts/AuthContext.tsx` (или аналог) для состояния пользователя и токена; логин/логаут и подстановка токена в API.
- **Типы:** `types/api.ts` (или `src/types/`) для типов запросов/ответов и домена.

**Правила:**
- Без прямого `fetch` к API из компонентов; использовать `lib/api.ts`.
- Окружение: только переменные `NEXT_PUBLIC_*`; без серверных секретов в клиентском коде.
- Echo опционален в рантайме; при необходимости feature-flag или проверка `NEXT_PUBLIC_REVERB_APP_KEY`.

### Направление зависимостей

| Слой | Может зависеть от |
|------|-------------------|
| app/ (страницы, лейауты) | components, lib, contexts, types |
| components | lib (api, echo), contexts, types; не от app/ |
| lib/, contexts/ | types; не от components или app/ |
| types | Ничего (нет рантайм-зависимостей) |

**Запрещено:** компоненты и lib не должны импортировать из `app/`. Страницы/лейауты не должны содержать бизнес-логику, принадлежащую lib или хукам.

## Паттерн взаимодействия с сервисами

1. **Действие пользователя** (напр. «Запустить анализ») → страница или контейнер вызывает функцию (из хука или lib).
2. **Вызов API** — функция использует `lib/api.ts` для POST/GET; токен подставляется API-клиентом из контекста авторизации.
3. **Реальное время (при необходимости)** — после запуска анализа клиент подписывается через `lib/echo.ts` на канал контракта; события прогресса обновляют локальное состояние.
4. **Обновление UI** — состояние хранится в состоянии компонента или контексте; компоненты перерисовываются; без прямых вызовов API внутри презентационных компонентов.

**Пример последовательности (поток анализа):**
```
Пользователь нажимает «Анализ» → страница вызывает api.analyseContract(text)
  → API возвращает id задачи/контракта
  → Страница подписывается на канал Echo для этого контракта
  → События прогресса обновляют состояние
  → По завершении страница загружает полный анализ через api.getContract(id) и т.п.
  → Компоненты отображают анализ из состояния
```

## Стратегия расширения

- **Новая страница под `/ai/`:** добавить папку в `app/ai/` с `page.tsx`; переиспользовать `app/ai/layout.tsx`; использовать существующий API-клиент и типы.
- **Новая поверхность API:** добавить методы в `lib/api.ts` и типы в `types/api.ts`; вызывать со страниц или из хуков.
- **Новое общее состояние:** предпочтительно React context в `contexts/`; при глобальном или долгоживущем — оформить в ADR.
- **Новый переиспользуемый UI:** добавлять в `components/ui/` или в папку фичи; без зависимости от маршрутов app.

## Запрещённые антипаттерны в проекте

| Антипаттерн | Правило |
|-------------|---------|
| Бизнес-логика в `page.tsx` | Страницы — только композиция; логика в lib/хуках |
| Прямой `fetch` к API | Только `lib/api.ts` |
| Глобальные изменяемые синглтоны | Использовать React state или context |
| Секреты в клиентском бандле | Только `NEXT_PUBLIC_*`; без серверных секретов |
| Обход авторизации на `/ai/*` | Использовать layout или ProtectedRoute |
| Импорт app из компонентов | Направление зависимостей одностороннее |
| Нетипизированные ответы API | Использовать типы из `types/api.ts` |
| Отключение ESLint/TypeScript без причины | Исправить или задокументировать исключение |
