# Карта контекстов — Contractor Frontend

## Ограниченные контексты

Фронтенд разделён на **ограниченные контексты** по зонам ответственности. У каждого контекста чёткие границы и точки интеграции; доступ между контекстами — по заданным контрактам.

### 1. Контекст публичной оболочки (Public Shell)

**Область:** Неавторизованный лендинг, маркетинг и юридические страницы.

**Ответственность:**
- Главная (`/`).
- Страницы входа и регистрации.
- Колбэк авторизации (OAuth redirect).
- Юридические: конфиденциальность, условия, cookie policy.
- Публичный хедер/футер и баннер cookie.

**Владение:** Маршруты в `app/` (кроме `app/ai/`, `app/auth/`, `app/payment/`); компоненты в `components/public/`, `HomeHeaderNav.tsx`, `GoogleAuthButton.tsx`, `CookieBanner.tsx` и т.д.

**Интеграция:**
- Использует **контекст авторизации** для логина/регистрации и редиректа после авторизации.
- Вызывает **Contractor API** для авторизации (login, register, callback); без эндпоинтов контрактов и анализа.

**Ограничения:**
- Не отображает оболочку и дашборд для авторизованных.
- Не обращается к Echo и эндпоинтам API, специфичным для контрактов.

---

### 2. Контекст авторизованного приложения (AI / Dashboard)

**Область:** Все потоки авторизованного пользователя: дашборд, анализ, просмотр контракта, токены, профиль.

**Ответственность:**
- Layout для авторизованных (AiShell, сайдбар, топбар).
- Дашборд: список контрактов, фильтры, empty state.
- Анализ: ввод/загрузка и запуск анализа; отображение прогресса и результатов.
- Детали контракта: полный анализ, секции, оценки.
- Токены: баланс и вход в покупку.
- Профиль: данные пользователя и действия с аккаунтом.

**Владение:** Всё в `app/ai/`; компоненты в `components/ai/`, `components/contract-analysis/`; использование `ProtectedRoute` или проверка авторизации на уровне layout.

**Интеграция:**
- **Контекст авторизации:** токен и пользователь; используется API-клиентом и для условного UI.
- **API-клиент (`lib/api.ts`):** все эндпоинты контрактов, анализа, токенов и профиля.
- **Echo (`lib/echo.ts`):** подписка на канал прогресса анализа контракта.
- **Stripe:** оформление покупки токенов (Stripe.js; оплату обрабатывает бэкенд).

**Ограничения:**
- Не должен отображаться без валидной авторизации; при неавторизованном — редирект на логин.
- Не должен вызывать API без подстановки токена (обеспечивается lib/api).

---

### 3. Контекст платёжного потока (Payment Flow)

**Область:** Вход в checkout и обработка возвратов success/cancel.

**Ответственность:**
- Переход пользователя в Stripe Checkout (или аналог) для покупки токенов.
- Страницы успеха/отмены (`/payment/success`, `/payment/cancel`) с ясными сообщениями и навигацией обратно в приложение.

**Владение:** Страницы `app/payment/`; компоненты, запускающие checkout (напр. со страницы токенов).

**Интеграция:**
- **API:** бэкенд отдаёт сессию или ссылку checkout; фронтенд делает редирект.
- **Авторизация:** пользователь должен быть залогинен для покупки; токен отправляется с запросом к API.
- **Stripe:** на клиенте только Stripe.js для редиректа или сессии; секретных ключей на фронтенде нет.

**Ограничения:**
- Обработка платёжных секретов и вебхуков — только на бэкенде.
- Страницы success/cancel не выполняют бизнес-логику помимо отображения и редиректа.

---

### 4. Общее ядро (Shared Kernel)

**Область:** Код, используемый несколькими контекстами и не принадлежащий одной фиче.

**Содержимое:**
- **lib/api.ts** — единый API-клиент; подстановка токена; базовый URL из env.
- **lib/echo.ts** — WebSocket и подписка на каналы.
- **contexts/AuthContext.tsx** — состояние пользователя и токена; логин/логаут.
- **types/api.ts** — общие типы для запросов/ответов API.
- **components/ui/** — кнопки, карточки, контейнеры, спиннеры, бейджи и т.д.

**Правила:**
- Общее ядро не зависит от конкретных маршрутов или фичевых компонентов.
- Контексты зависят от общего ядра; общее ядро не зависит от контекстов.
- Изменения API или Echo затрагивают все потребляющие контексты; согласовывать и документировать.

## Матрица ответственности контекстов

| Контекст | Авторизация | API (контракт/анализ) | Echo | Stripe (клиент) | Маршруты |
|----------|-------------|------------------------|------|-----------------|----------|
| Public Shell | Только логин/регистрация | Только эндпоинты авторизации | Нет | Нет | `/`, `/login`, `/register`, `/auth/callback`, `/privacy`, `/terms`, `/cookie-policy` |
| Authenticated App | Обязательна | Все контракт/токен/профиль | Да (прогресс) | Вход в checkout | `/ai/*` |
| Payment Flow | Обязательна | Сессия/ссылка checkout | Нет | Редирект/сессия | `/payment/success`, `/payment/cancel` |
| Shared Kernel | Предоставляет токен/пользователя | Предоставляет клиент | Предоставляет клиент | — | — |

## Контракты интеграции

### Фронтенд ↔ Contractor API

- **Протокол:** REST по HTTPS; JSON запрос/ответ.
- **Авторизация:** Bearer JWT в заголовке `Authorization`; токен из контекста авторизации.
- **Базовый URL:** `NEXT_PUBLIC_API_URL`; без завершающего слэша в конфиге.
- **Ошибки:** фронтенд трактует не-2xx как ошибки; при наличии — сообщение или код от API; без тихих сбоев.

### Фронтенд ↔ Laravel Reverb (Echo)

- **Назначение:** прогресс анализа контракта в реальном времени.
- **Подключение:** Laravel Echo + протокол Pusher; конфиг из `NEXT_PUBLIC_REVERB_*`.
- **Каналы:** на контракт или на пользователя по определению API; фронтенд подписывается после старта анализа.
- **Резерв:** при недоступности Echo фронтенд может опрашивать API или показывать «Обработка…» до завершения по API.

### Фронтенд ↔ Stripe

- **Клиент:** Stripe.js (только publishable key); редирект в Checkout или session ID от API.
- **Секреты:** никогда на фронтенде; создание платежа и вебхуки на бэкенде.

## Обзор потока данных

1. **Публичная зона:** пользователь на `/` или `/login` → через контекст авторизации логинится → API возвращает токен → контекст сохраняет токен → редирект на `/ai/dashboard`.
2. **Дашборд:** layout проверяет авторизацию → страница дашборда загружается → `api` запрашивает список контрактов → компоненты отображают список и фильтры.
3. **Анализ:** пользователь отправляет текст → `api` запускает анализ → подписка на Echo для прогресса → обновление состояния прогресса в UI → по завершении загрузка анализа и отображение в компонентах contract-analysis.
4. **Оплата:** пользователь нажимает «Купить токены» → `api` создаёт сессию checkout → редирект в Stripe → возврат на `/payment/success` или `/payment/cancel` → ссылка обратно на `/ai/tokens` или дашборд.

---

## Ограничения между контекстами

- **Public Shell** не должен импортировать компоненты только для авторизованного приложения и не должен вызывать API контрактов/анализа.
- **Authenticated App** не реализует UI логина/регистрации (это в Public Shell); только читает состояние авторизации и редиректит при отсутствии логина.
- **Payment Flow** не содержит бизнес-логику анализа контрактов или Echo; только редирект и UI success/cancel.
- **Shared Kernel** не зависит от структуры маршрутов и фичевых компонентов; предоставляет общие API, Echo, авторизацию и UI-примитивы.

Эта карта контекстов — единственный источник истины для границ; перенос ответственности между контекстами требует ADR.
