# Безопасность — Contractor Frontend

## Область применения

В документе описаны практики безопасности для Contractor Frontend: что фронтенд должен и не должен делать, как обрабатывать чувствительные данные и как безопасно интегрироваться с API и сторонними сервисами.

## Принципы

1. **Никаких секретов на клиенте.** В браузер попадают только переменные `NEXT_PUBLIC_*`; они не должны содержать секреты API, секретный ключ Stripe или секрет приложения Reverb.
2. **Авторизация — зона ответственности API.** Фронтенд хранит и отправляет токены по правилам API (напр. JWT в localStorage и заголовок `Authorization`); логику авторизации и хранение паролей не реализует.
3. **Минимизировать поверхность атаки.** Валидировать и санитизировать только необходимое; полагаться на API в вопросах валидации и авторизации.
4. **Безопасность по умолчанию.** В продакшене использовать HTTPS, при наличии cookies — защищённые; не допускать попадания токенов в URL или логи.

## Обработка чувствительных данных

### Токены и учётные данные

| Данные | Правило |
|--------|---------|
| JWT / access token | Хранить по контракту API (напр. localStorage); отправлять только в заголовке `Authorization`; никогда в query/path URL или в логах. |
| Refresh token | При использовании — как JWT; не раскрывать в клиентских логах или отчётах об ошибках. |
| Секретный ключ Stripe | Никогда на фронтенде; только на бэкенде. |
| Секрет приложения Reverb | Только на бэкенде. |
| API-ключи (серверные) | На фронтенде не используются. |

### Персональные и пользовательские данные

| Данные | Правило |
|--------|---------|
| Email, имя | Показывать только где нужно; не логировать; передавать в API по HTTPS. |
| Содержимое контракта | Отправлять в API по HTTPS; не логировать полный текст; не хранить на клиенте сверх необходимого для текущего экрана. |
| Платёжные данные | На фронтенде не обрабатываются; карточные данные — через Stripe.js; в нашем коде и логах данных карт нет. |

### Плейсхолдеры в коде и документации

- Использовать явные плейсхолдеры (напр. `YOUR_STRIPE_PUBLISHABLE_KEY`, `NEXT_PUBLIC_API_URL`).
- В документации напоминать о замене плейсхолдеров и о недопустимости коммита реальных значений.
- Секреты задавать через окружение или защищённый конфиг, не хардкодить.

## Безопасные практики

### Аутентификация и авторизация

- **Защищённые маршруты:** все `/ai/*` и платёжные потоки требуют валидной авторизации; использовать layout или `ProtectedRoute` и редиректить неавторизованных на логин.
- **Подстановка токена:** один API-клиент (`lib/api.ts`), берущий токен из одного источника (контекст авторизации или localStorage); избегать разрозненной работы с токеном.
- **Выход:** очищать токен и состояние пользователя; редирект на публичную главную или логин.

### Взаимодействие с API

- **HTTPS:** в продакшене и на стейджинге приложение и `NEXT_PUBLIC_API_URL` должны использовать HTTPS.
- **CORS:** API разрешает только нужные origins; фронтенд не ослабляет CORS.
- **Ошибки:** не показывать пользователю стек и внутренние детали ошибок сверх общего сообщения; детали логировать только в защищённом или контролируемом канале отчётов.

### Ввод и вывод

- **XSS:** React по умолчанию экранирует текст; для markdown или HTML использовать санитизированный путь рендера (напр. `MarkdownContent` с безопасным подмножеством).
- **Инъекции:** не собирать URL или запросы из несанитизированного ввода; использовать типизированные методы API и при необходимости кодировать.
- **Загрузка файлов:** при отправке файлов с фронтенда проверять тип/размер на клиенте только для UX; лимиты и проверка содержимого — на стороне API.

### Сторонние скрипты

- **Stripe:** только официальный Stripe.js; загружать с домена Stripe; только publishable key.
- **Аналитика и прочие скрипты:** предпочтительно same-origin или доверенные CDN; избегать инлайн-скриптов с пользовательскими или чувствительными данными.

## Чего фронтенд не должен делать

- Хранить или отправлять секретные ключи API, секретный ключ Stripe или секрет приложения Reverb.
- Реализовывать хеширование паролей или выдачу токенов; это зона API.
- Обходить защищённые маршруты или отправлять запросы без требуемой авторизации при залогиненном пользователе.
- Логировать токены, пароли или полный текст контракта.
- Отключать или обходить HTTPS в продакшене.
- Показывать конечному пользователю детальные серверные ошибки или стеки.

## При обнаружении уязвимости

- Не коммитить и не обсуждать проблему в публичных каналах.
- Сообщить владельцу проекта или по процессу, определённому для репозитория (напр. приватный отчёт, security-email).
- Не эксплуатировать сверх необходимого для демонстрации проблемы.

## Чек-лист для новых фич

| Пункт | Проверить |
|-------|-----------|
| Нет новых секретов на клиенте | Только `NEXT_PUBLIC_*` или несекретная конфигурация |
| Авторизация на защищённых маршрутах | Layout или guard для `/ai/*` и оплаты |
| Токен не в URL и не в логах | Токен только в заголовке и защищённом хранилище |
| Пользовательские/чувствительные данные не логируются | Удалять или маскировать в логах и отчётах об ошибках |
| Markdown/HTML санитизированы | Использовать существующий безопасный путь рендера |
| HTTPS в продакшене | Окружение и деплой используют HTTPS и WSS |
