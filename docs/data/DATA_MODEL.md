# Модель данных — Contractor Frontend

## Назначение

В документе описаны **клиентские** структуры данных и контракты API, используемые Contractor Frontend. Источник истины по хранению и бизнес-правилам — Contractor API; фронтенд потребляет эти структуры для отображения и отправки форм.

## Поток данных

1. **Сервер → Клиент:** API возвращает JSON; фронтенд типизирует его (напр. в `src/types/api.ts`) и использует в компонентах.
2. **Клиент → Сервер:** фронтенд отправляет JSON (и токен авторизации) через `lib/api.ts`; тела запросов соответствуют ожиданиям API.
3. **Реальное время:** Echo доставляет события (напр. прогресс анализа); фронтенд маппит события в локальное состояние и UI.

Фронтенд не владеет канонической схемой; он согласуется с API. При изменении API обновлять типы и затронутые компоненты.

## Основные типы и структуры

### Контракт (кратко и детально)

Используется в списках и детальном просмотре. Согласовать с ответом API; типичные поля:

| Поле | Тип | Примечания |
|------|-----|------------|
| `id` | string | Уникальный идентификатор |
| `title` | string \| null | Название для пользователя |
| `status` | string | напр. `pending`, `processing`, `completed`, `failed` |
| `created_at` | string (ISO дата) | Для сортировки и отображения |
| `analysis` | object \| null | Присутствует при status `completed`; структура ниже |
| (другие) | — | По API (напр. `user_id`, `updated_at`) |

**Пример (минимальный):**
```ts
interface Contract {
  id: string;
  title: string | null;
  status: "pending" | "processing" | "completed" | "failed";
  created_at: string;
  analysis?: AnalysisResult | null;
}
```

### Результат анализа

Вложен в контракт при завершённом анализе. Структура должна совпадать с API (секции, оценки, саммари).

| Концепт | Расположение | Примечания |
|---------|--------------|------------|
| Summary / quick summary | `analysis.summary` или аналог | Короткий текст или структура |
| Секции | `analysis.sections` или аналог | Массив блоков секций |
| Оценки / риски | `analysis.scores`, `analysis.risks` или по секциям | Для бейджей и карточек оценок |
| Сырой markdown | По секции или глобально | Рендер через `MarkdownContent` |

Точные интерфейсы задавать в `src/types/api.ts` по реальному ответу API.

### Пользователь и авторизация

| Концепт | Источник | Примечания |
|---------|----------|------------|
| Данные пользователя | API (напр. `/user`, `/me`) или payload токена | Имя, email, принятие условий |
| Токен | API (login/register/callback) | Хранится по текущей схеме (напр. localStorage); в типах не как «модель», а как непрозрачная строка в контексте авторизации |

### Токены (кредиты) и оплата

| Концепт | Источник | Примечания |
|---------|----------|------------|
| Баланс | API (напр. `/tokens` или payload пользователя) | Число или объект с `balance` |
| Checkout | API возвращает URL или ID сессии | Фронтенд редиректит в Stripe; данных платёжного метода в наших типах нет |

## Контракт API-клиента

- **Базовый URL:** `NEXT_PUBLIC_API_URL` (без завершающего слэша).
- **Авторизация:** `Authorization: Bearer <token>` для защищённых эндпоинтов; токен из контекста авторизации или localStorage.
- **Запрос:** при необходимости тело JSON; `Content-Type: application/json`.
- **Ответ:** JSON; 2xx — успех, 4xx/5xx — ошибки с телом (напр. `{ error: string, code?: string }`).
- **Ошибки:** выбрасываются `lib/api.ts` с сообщением и опционально `code`; типизировать как `ApiError` или аналог.

## События в реальном времени (Echo)

- **Канал:** на контракт или на пользователя по определению API.
- **Payload события:** напр. `{ stage: string, progress?: number, message?: string }` для прогресса; завершение может быть событием или выводиться из опроса API.
- **Фронтенд:** подписаться после старта анализа; обновлять локальное состояние; отписаться при уходе или завершении.

## Где хранятся типы

| Расположение | Назначение |
|--------------|------------|
| `src/types/api.ts` | Типы запросов/ответов API, `ApiError`, общие доменные типы |
| Файлы компонентов | Интерфейсы пропсов (напр. `ContractSummaryCardProps`) при отсутствии переиспользования |
| `lib/api.ts` | Экспорт `ApiError` и общих типов запросов/ответов |

## Правила и запреты

- **Без дублирования определений:** один канонический тип на концепт (предпочтительно `types/api.ts` для форм API).
- **Без персистентности на клиенте:** не вводить клиентские «модели БД»; данные считаются эфемерными и подгружаются с API.
- **Nullable и optional:** использовать `| null` или `?` в соответствии с API; в UI обрабатывать отсутствующие поля (напр. «Без названия», 0 или скрывать).
- **Даты:** хранить как ISO-строки от API; форматировать для отображения в компонентах или небольшой утилитой; семантику не менять.

При добавлении или изменении полей в API обновлять этот документ и `src/types/api.ts` вместе; при пользовательских изменениях — краткая заметка в CHANGELOG или PR.
