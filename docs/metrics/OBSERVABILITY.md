# Наблюдаемость — Contractor Frontend

## Назначение

В документе определено, как Contractor Frontend ведёт логирование, отчётность об ошибках и мониторинг, чтобы проблемы можно было диагностировать и исправлять без раскрытия чувствительных данных.

## Логирование

### Что логировать

| Окружение | Разрешено | Запрещено |
|-----------|-----------|-----------|
| Разработка | Отладочные логи, предупреждения, ошибки API (с осторожностью) | Токены, пароли, полный текст контракта |
| Продакшен | Высокоуровневые события (напр. «анализ запущен», «редирект на оплату»), коды или сообщения об ошибках (без PII) | Токены, пароли, PII, полные тела запросов/ответов |

### Куда писать логи

- **Разработка:** допускаются `console.log` / `console.warn` / `console.error` для локальной отладки.
- **Продакшен:** предпочтительнее один канал: либо сервис отчётов об ошибках (напр. Sentry), либо структурированные логи на бэкенд; не разбрасывать `console` для эксплуатационных целей.
- **Без секретов:** никогда не логировать заголовок `Authorization`, токен, ключи Stripe или секреты Reverb. В продакшен-логах маскировать или не включать PII.

### Уровни логов

При отсутствии библиотеки логгера использовать простое соглашение:

- **Error:** необработанные исключения, неудачные вызовы API, влияющие на пользователя, сбои подключения Echo.
- **Warn:** восстанавливаемые ситуации (напр. переход с Echo на опрос, отсутствие опциональной переменной окружения).
- **Info/Debug:** использование фич или шаги потоков только в разработке или при явном включении.

## Отчётность об ошибках

### Клиентские ошибки

- **Необработанные исключения:** отправлять в сервис отчётов (напр. Sentry) с контекстом: сообщение, стек, компонент или маршрут; не прикладывать токен или PII.
- **Ошибки API:** либо отправлять только код статуса и эндпоинт (и при необходимости безопасное сообщение), либо давать инструменту захватывать объект ошибки после удаления чувствительных полей.
- **Обратная связь пользователю:** показывать общее сообщение и при необходимости кнопку повтора; не показывать стеки и внутренние детали.

### Интеграция

- **Конфиг:** использовать переменные окружения (напр. `NEXT_PUBLIC_SENTRY_DSN`), чтобы одна сборка могла работать с разными окружениями.
- **Source maps:** загружать в сервис ошибок для продакшена (по документации вендора); не отдавать source maps публично, если в них чувствительные пути или логика.
- **Сэмплирование:** учитывать ограничение частоты или сэмплирование в продакшене для контроля затрат и шума.

## Мониторинг

### Метрики для учёта

| Метрика | Назначение |
|---------|------------|
| Загрузка страницы / LCP | Core Web Vitals |
| Длительность и статус вызовов API | Выявление замедлений или сбоев API |
| Статус подключения Echo | Надёжность в реальном времени |
| Клиентские ошибки (количество по типу) | Тренды и алерты по регрессиям |

Реализация — через сервис ошибок, RUM-продукт или лёгкий эндпоинт на бэкенде для анонимизированных метрик. Не отправлять PII или токены.

### Проверки здоровья

- **Сборка:** в CI выполняются `npm run build` и `npm run lint` (см. DEPLOYMENT).
- **Рантайм:** корневой URL возвращает 200; опционально маршрут «ping» или «health», вызывающий безопасный эндпоинт API и возвращающий статус для мониторинга.

## Правила и запреты

- **Никогда не логировать и не отправлять в отчёты:** токены, пароли, секреты API, ключи Stripe, полный текст контракта или иной PII, если только это явно не требуется и надёжно защищено.
- **Продакшен:** предпочтительны структурированные, минимальные логи и отчёты об ошибках; избегать подробного вывода в консоль.
- **Согласие:** при требовании регуляторов согласия на необязательный трекинг или отчётность — документировать и реализовать соответственно.

## Пример: Error boundary и отчётность

```tsx
// Концептуально: отправить ошибку без PII
function reportError(error: Error, context?: Record<string, string>) {
  if (process.env.NEXT_PUBLIC_SENTRY_DSN) {
    Sentry.captureException(error, { extra: context });
  }
}
```

Использовать в error boundaries и критичных catch; в `context` не передавать токены и PII.
