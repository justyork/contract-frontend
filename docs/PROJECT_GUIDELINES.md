# Руководство по проекту — Contractor Frontend

## Цели проекта

1. **Реализовать UI Contralytic MVP** — опыт в стиле SPA: загрузка, анализ контрактов, просмотр результатов, управление токенами и оплатой.
2. **Надёжно потреблять Contractor API** — всё серверное состояние и действия идут через API; фронтенд остаётся без состояния, кроме сессии и UI.
3. **Обратная связь в реальном времени** — Laravel Echo (Reverb) для прогресса анализа и будущих обновлений.
4. **Поддерживаемость** — чёткая структура (Next.js App Router, типизированный API, общие компоненты) для добавления фич без поломки существующих потоков.
5. **Документация для AI** — документация и ADR поддерживают и людей, и разработку с помощью агентов.

## Вне рамок (не цели фронтенда)

- Реализация анализа контрактов или AI-логики (только бэкенд).
- Хранение или обработка документов на фронтенде сверх загрузки и отображения.
- Замена или дублирование авторизации API; используем JWT и OAuth Contractor API.
- Создание дизайн-системы или библиотеки компонентов для других продуктов; рамки — это приложение.

## Критерий «Готово»

Фича или изменение считаются готовыми, когда:

| Критерий | Требование |
|----------|-------------|
| Функциональность | Работает в браузере по счастливому пути и заданным граничным случаям |
| Использование API | Используется `src/lib/api.ts` и задокументированные эндпоинты; без ad-hoc fetch |
| Типы | Типы TypeScript для ответов API и ключевых пропсов; без `any` для публичного API |
| Маршруты | Соответствие текущей структуре маршрутов/layout или наличие ADR для новой |
| Авторизация | Защищённые маршруты используют текущую авторизацию (напр. ProtectedRoute / layout) |
| Ошибки | Реализованы состояния загрузки и ошибки; без тихих сбоев |
| Тесты | Новая логика покрыта unit- или интеграционными тестами по TESTING.md |
| Документация | README или setup обновлены при изменении env или инструкций запуска |
| Линтер | ESLint и TypeScript проходят; правила не отключаются без обоснования |

## Ожидания по масштабированию

- **Пользователи:** фронтенд без состояния; масштабирование обеспечивают API и хостинг (напр. Vercel).
- **Параллельные анализы:** прогресс доставляется по WebSocket; UI только подписывается и обновляется.
- **Бандл:** контролировать размер клиентского бандла; при необходимости лениво подгружать тяжёлые маршруты или компоненты.
- **Данные:** без крупных клиентских кэшей; списки и детали с API; пагинация при росте списков.

## Базовые показатели производительности

- **LCP:** цель — менее 2,5 с на быстром 3G для критического пути (логин → дашборд или анализ).
- **Взаимодействие:** кнопки и навигация без заметной задержки; для асинхронных операций — состояния загрузки.
- **Сборка:** продакшен-сборка завершается успешно; без жёстких ошибок при сборке и проверке типов.
- **Запрет:** без блокирующей синхронной работы в main thread для больших данных; использовать стриминг или пагинацию.

## Базовый уровень безопасности

- **Секреты:** без секретов API и секретных ключей Stripe на фронтенде; в клиентский бандл только `NEXT_PUBLIC_*`.
- **Авторизация:** JWT от API хранится и отправляется по текущей схеме (напр. localStorage + API-клиент); потоки авторизации не должны раскрывать токены в URL или логах.
- **Ввод:** пользовательский ввод в UI экранируется React; markdown или богатый контент санитизируется по использованию MarkdownContent.
- **HTTPS:** продакшен и стейджинг по HTTPS; WebSocket по WSS в продакшене.
- **CORS:** API вызывается с того же origin или разрешённых origins; без широкого `*` в продакшене.

## Правила владения кодом

- **Репозиторий/приложение фронтенда** — этот кодбаз владеет всем UI, клиентским состоянием и использованием API-клиента для фронтенда Contralytic.
- **Контракт API** — владелец Contractor API; фронтенд следует контракту и сообщает о нарушениях; изменения согласуются.
- **Дизайн и тексты** — следовать текущим паттернам и юридическим страницам (конфиденциальность, условия, cookie); существенные изменения текстов — продуктовые/юридические решения.

## Правила управления изменениями

1. **Маршруты и layout:** новые маршруты верхнего уровня или изменения layout требуют ревью; существенные изменения — ADR.
2. **API-клиент или Echo:** изменения в `lib/api.ts` или `lib/echo.ts` документируются и тестируются; изменения контракта — с командой API.
3. **Зависимости:** новые рантайм-зависимости — краткое обоснование; крупные обновления (Next, React, TypeScript) — ревью и тесты.
4. **Окружение:** новые переменные `NEXT_PUBLIC_*` добавляются в `.env.example` и ENVIRONMENT.md.
5. **Документация:** ссылки в AGENTS.md и docs остаются актуальными при изменении структуры или процесса.

## Пример: в рамках vs вне рамок

| В рамках | Вне рамок |
|----------|-----------|
| Новый фильтр или empty state на дашборде | Новый алгоритм анализа на бэкенде |
| Новая страница под `/ai/*` в текущем layout | Новый провайдер авторизации без поддержки API |
| Рефакторинг компонентов для переиспользования | Создание общей дизайн-системы |
| Добавление E2E для критических потоков | Нагрузочное тестирование API |
| Обновление документации по env | Изменение деплоя API |

Эти руководства — ориентир для приоритизации и для AI-агентов при решении, что относится к этому проекту.
