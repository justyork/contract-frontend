# Конфигурация AI-агентов для Contractor Frontend

## Назначение

В этом документе определены роли, обязанности и рабочие процессы для AI-агентов, работающих над проектом Contractor Frontend. Contractor Frontend — приложение на Next.js 16 (App Router) с React 19 и TypeScript, интерфейс для Contralytic (SaaS анализа контрактов на базе ИИ). Оно потребляет Contractor API и использует Laravel Echo для обновлений прогресса в реальном времени.

## Роли

### Архитектор

**Обязанности:**
- Определять архитектурные паттерны и проектные решения для фронтенда
- Проверять и утверждать структурные изменения (маршруты, поток данных, состояние)
- Создавать и поддерживать ADR по архитектуре фронтенда
- Обеспечивать соблюдение практик Next.js App Router и React
- Проверять границы страниц/компонентов и направление зависимостей
- Утверждать новый глобальный стейт, контекст или общие библиотеки

**Границы:**
- Не изменяет код реализации напрямую
- Должен фиксировать все архитектурные решения в ADR
- Не может обходить установленные паттерны без обоснования

### Разработчик

**Обязанности:**
- Реализовывать функции в заданных архитектурных границах
- Писать код по соглашениям Next.js и TypeScript
- Создавать и поддерживать компоненты, страницы и хуки
- Корректно интегрироваться с API и WebSocket (Echo)
- Обеспечивать доступность и адаптивное поведение
- Соблюдать стиль кода и требования к тестам проекта

**Границы:**
- Не добавляет маршруты верхнего уровня и не меняет layout без одобрения Архитектора
- Не вводит новый глобальный стейт/контекст без ревью
- Должен следовать существующим паттернам в `src/`
- Не может добавлять зависимости без ревью

### Ревьюер

**Обязанности:**
- Ревьюить код на качество, паттерны и лучшие практики
- Проверять соблюдение соглашений Next.js и TypeScript
- Проверять структуру и переиспользуемость компонентов
- Проверять использование API/контекста и обработку ошибок
- Обеспечивать доступность и единообразие UX

**Границы:**
- Не утверждает архитектурные изменения (это роль Архитектора)
- Не правит код напрямую; запрашивает изменения у Разработчика
- Должен давать конкретные замечания

### QA

**Обязанности:**
- Создавать и поддерживать тесты фронтенда (unit, интеграция, E2E)
- Проверять работу функций и поведение UI
- Тестировать интеграцию с API и WebSocket
- Проверять адаптивность и доступность

**Границы:**
- Не изменяет продакшен-код
- Должен использовать существующую тестовую инфраструктуру
- Не может обходить требования к тестам

### DevOps

**Обязанности:**
- Управлять конфигурацией сборки и развёртывания
- Поддерживать окружение и CI/CD для фронтенда
- Настраивать превью и продакшен-деплой
- Мониторить производительность и ошибки фронтенда

**Границы:**
- Не меняет логику приложения и UI-код
- Должен следовать установленным практикам развёртывания
- Не может менять требования к окружению без одобрения

## Правила генерации кода

### Структура каталогов

Код должен следовать принятой структуре Next.js App Router:

```
src/
├── app/                    # Маршруты, лейауты, страницы
│   ├── layout.tsx          # Корневой лейаут
│   ├── page.tsx            # Главная
│   ├── ai/                 # Авторизованное приложение (дашборд, анализ и т.д.)
│   ├── auth/               # Колбэк авторизации
│   ├── login/, register/
│   ├── payment/            # Успех/отмена
│   └── privacy/, terms/, cookie-policy/
├── components/             # Переиспользуемый UI
│   ├── ai/                 # Компоненты оболочки и дашборда
│   ├── contract-analysis/  # Компоненты отображения анализа
│   ├── public/             # Оболочка/футер публичных страниц
│   └── ui/                 # Примитивы (Button, Card и т.д.)
├── contexts/               # React-контекст (напр. AuthContext)
├── lib/                    # API-клиент, Echo, утилиты
└── types/                  # Общие типы TypeScript
```

### Соглашения по именованию

- **Компоненты:** PascalCase, описательные (напр. `ContractSummaryCard`, `AiShell`)
- **Страницы:** папки маршрутов в нижнем регистре; компонент в `page.tsx`
- **Хуки:** префикс `use` (напр. `useAuth`, `useContract`)
- **Утилиты:** camelCase (напр. `api.ts`, `echo.ts`)
- **Типы:** PascalCase, в `src/types/` или рядом с кодом

### Правила организации кода

1. **Границы маршрутов:** страницы в `app/` владеют своим маршрутом; общий layout в `app/ai/layout.tsx`.
2. **Без бизнес-логики в страницах:** страницы собирают компоненты и вызывают хуки/lib; логика в хуках, lib или компонентах.
3. **Доступ к API:** все вызовы API через `src/lib/api.ts`; не разбрасывать `fetch` с сырыми URL.
4. **Окружение:** использовать только переменные `NEXT_PUBLIC_*`; не раскрывать серверные секреты на клиенте.
5. **Защищённые маршруты:** использовать `ProtectedRoute` или проверку авторизации на уровне layout для `/ai/*`.

## Правила изменения архитектуры

### Новые маршруты или лейауты

1. Архитектор описывает в ADR, если это затрагивает навигацию или поток данных.
2. Определить, какой layout использует маршрут и требования к авторизации.
3. Разработчик реализует в рамках этого контракта.

### Новый глобальный стейт или контекст

1. Архитектор одобряет необходимость и объём.
2. Задокументировать, что глобально, а что локально.
3. Предпочтительно React context или состояние в URL; избегать произвольных глобальных переменных.

### Изменения контракта API

1. Согласовать с бэкендом (Contractor API); при изменении контракта фронтенда — оформить ADR.
2. Обновить `src/lib/api.ts` и `src/types/api.ts`.
3. Обновить затронутые компоненты и тесты.

## Правила обновления документации

- **ADR:** создаются при изменениях маршрутизации, стейта или паттернов интеграции.
- **README / docs/setup:** обновляются при изменении настройки или переменных окружения.
- **Комментарии в коде:** обновляются при изменении неочевидной логики; использовать декларативные комментарии в настоящем времени.

## Политика ADR

**Требуется для:** новых маршрутов или лейаутов верхнего уровня, нового глобального контекста/стейта, изменения паттерна использования API-клиента или Echo, нового потока авторизации, крупной зависимости (напр. библиотеки стейта).

**Процесс:** создать ADR в `docs/adr/` с контекстом, решением и последствиями; пройти ревью; пометить Accepted/Rejected/Superseded; при необходимости ссылаться в коде.

## Запрещённые действия

- **Архитектура:** циклические зависимости между модулями; бизнес-логика в `page.tsx` помимо композиции; прямой `fetch` к API в обход `lib/api.ts`.
- **Качество кода:** отключение ESLint без обоснования; хардкод переменных окружения; игнорирование ошибок TypeScript.
- **Безопасность:** хранение токенов вне httpOnly/защищённых схем; передача секретов в клиентский бандл; обход авторизации на защищённых маршрутах.
- **UX:** отсутствие состояний загрузки и ошибки; игнорирование доступности (подписи, фокус, семантика).

## Правила эскалации

К Архитектору: конфликт паттернов, новый глобальный стейт, переработка маршрутов/layout. К DevOps: сбои сборки/деплоя, окружение или хостинг. Проблемы безопасности эскалируются немедленно.

## Пример жизненного цикла фичи

1. **Планирование (Архитектор):** при необходимости ADR; определить маршрут, поток данных и использование API.
2. **Реализация (Разработчик):** фича-ветка; реализация в `src/`; использование `lib/api.ts` и типов; добавление тестов.
3. **Ревью (Ревьюер):** код-ревью по паттернам, типам и UX.
4. **Тестирование (QA):** запуск тестов; проверка поведения API и Echo.
5. **Деплой (DevOps):** сборка и развёртывание; проверка окружения и здоровья.

## Соответствие типов задач и ролей агентов

| Тип задачи | Основная роль | Вспомогательная роль | Примечания |
|------------|---------------|------------------------|------------|
| Новый маршрут или layout | Архитектор | Разработчик | ADR при структурных изменениях |
| Компонент или фича страницы | Разработчик | Ревьюер | Следовать существующим паттернам |
| Исправление бага | Разработчик | Ревьюер | Без новых паттернов |
| Изменение интеграции API или Echo | Разработчик | Архитектор | Документировать контракт |
| Новый глобальный контекст/стейт | Архитектор | Разработчик | Требуется ADR |
| Создание тестов | QA | Разработчик | Использовать тестовую настройку проекта |
| Обновление документации | Разработчик | Ревьюер | При изменении поведения |
| Конфиг сборки/деплоя | DevOps | Разработчик | Только окружение и CI |
| Обновление зависимостей | Разработчик | Архитектор | Крупные — с одобрением |
| Изменение авторизации или безопасности | Архитектор | Разработчик | Критично для безопасности |
